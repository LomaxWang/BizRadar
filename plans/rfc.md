# 📄 产品需求文档 (PRD)：Micro-SaaS 商业机会挖掘 Agent

**项目代号：** IdeaHunter / 商业雷达 Agent
**定位：** 专为中国独立开发者和微型技术团队打造的“赛博产品经理”与“商业机会捕手”。

## 一、 产品概述 (Product Overview)

### 1.1 项目背景
随着大模型 API 的普及，软件开发门槛大幅降低（“怎么做”不再是核心壁垒）。当前独立开发者和技术极客面临的最大痛点是**“缺乏商业嗅觉，不知道做什么能赚钱”**。开发者习惯于闭门造车，而真实世界中愿意付费的痛点往往深藏在下沉市场和异业圈层中。

### 1.2 产品愿景
打造一个 24 小时全天候运行的自动化商业雷达。通过多智能体协作（Multi-Agent），打通“数据获取 -> 需求过滤 -> 商业评估 -> 产品立项”的全链路，帮助技术人员跨越信息茧房，挖掘高胜率、高 ROI 的微小产品需求（Micro-SaaS）。

### 1.3 目标用户 (Target Audience)
*   **核心用户：** 寻找第一桶金的独立开发者 (Indie Hackers)、业余时间做副业的程序员。
*   **辐射用户：** 小型创业团队产品经理、寻找细分赛道的 SaaS 投资人/孵化器。

## 二、 核心评估漏斗（“好需求”判定黄金三定律）

IdeaHunter Agent 区别于普通舆情爬虫的核心，在于其内置了极其苛刻的**商业评估引擎 (Evaluation Matrix)**。任何抓取到的用户抱怨，必须通过以下三层漏斗的交叉验证：

1.  **高频体能测试 (High Frequency & Repetitive)：** 该痛点是否为每日发生？是否属于高密度的“复制、粘贴、搬运、格式转换”等机械劳动？（*滤除伪需求与极低频需求*）
2.  **大厂生态免疫测试 (Platform Gap & Anti-Monopoly)：** 该问题是否因为各大平台（如微信、抖音、淘宝）互设护城河产生？是否属于大厂“看不上”或“官方不方便直接做”的灰色效率地带？（*确保生存空间*）
3.  **商业闭环测试 (Clear ROI)：** 解决该问题后，能否直接帮目标用户“省出时间去接更多单/卖更多货”？用户是否具有明确的“买工具抵扣人工费”的付费习惯？（*确保能够变现*）

## 三、 核心监控场景与业务矩阵 (Key Scenarios)

基于国内真实商业环境，Agent 初始版本将重点锚定以下四大“离钱最近”的赛道进行数据挖掘：

| 业务赛道 | 典型监控人群 | 核心痛点特征 (Agent 捕获关键词) | 潜在输出的产品形态示例 |
| :--- | :--- | :--- | :--- |
| **1. 流量倒卖与自媒体矩阵** | 小红书博主、短视频矩阵号、小说推文团队 | 防判定查重、跨平台搬运、无版权素材查找耗时 | 一键仿写+自动配图生成器、跨平台视频去水印/切片工具 |
| **2. 非对称电商与二道贩子** | 闲鱼无货源卖家、拼多多店群玩家、跨境独立站站长 | 人工跨平台比价眼盲、海量商品主图去背景/加边框 | 跨平台价格监控浏览器插件、批量商品图“白底化+加框”脚本 |
| **3. 教育焦虑与知识提纯** | 考研/考公党、各类资格证备考者 | 非结构化资料堆积（乱码PDF/群截图）、死记硬背效率低 | 微信群截图一键转 Anki 记忆卡片生成器、公考错题本整理助手 |
| **4. 传统行业私域运营** | 房产中介、保险代理人、微商、线下教培 | 朋友圈发图易折叠、重复客诉沟通内耗高、排版费时 | 房产中介专属防折叠排版工具、高情商/定制化销售回复输入法 |

## 四、 核心系统架构与工作流 (System Architecture & Workflow)

结合业界优秀的 Multi-Agent 实践（如 LangGraph/AutoGen 的编排理念），IdeaHunter 采用**“三段式流水线 + 评审委员会”**架构：

### 阶段一：感知层 (Sensor Layer) -> `Scraper Agent`
*   **职责：** 充当雷达。通过轻量级爬虫或 API，定时巡航指定目标源（如 V2EX 特定节点、小红书特定话题搜索、知乎问题、特定行业公众号文章）。
*   **最佳实践：** 不做全网无脑抓取，采用**定向关键词触发**（如：“求推荐工具”、“太痛苦了”、“每天都要手动”）。

### 阶段二：思考层 (Cognitive Layer) -> `The Board (评审委员会)`
这是整个系统的灵魂，采用**多角色辩论模式 (Debate Mechanism)**，包含三个子 Agent：
1.  **产品经理 Agent (The PM)：** 负责将抓取到的非结构化抱怨，翻译成清晰的用户故事（User Story）。
2.  **毒舌投资人 Agent (The Critic)：** 严格套用“黄金三定律”进行打分。会主动检索市面上是否已有成熟竞品，若有竞品且口碑良好，则直接毙掉该想法。
3.  **技术合伙人 Agent (The Tech Lead)：** 评估实现成本。如果判定该需求需要训练底层大模型或开发周期超过 2 个月，则降低推荐权重；偏好 API 拼接和轻量级前后端方案。

### 阶段三：输出层 (Action Layer) -> `Planner Agent`
*   **职责：** 汇总是委员会的共识，输出结构化的行动指南。
*   **输出物标准格式：《微型产品立项书》**
    *   发现的痛点及原始数据出处。
    *   目标用户画像与付费意愿评估。
    *   市面竞品分析（如果有）。
    *   MVP（最小可行性产品）核心功能清单。
    *   推荐的技术栈（如：FastAPI + Vue + Kimi API）。
    *   冷启动获客渠道建议。

## 五、 项目演进路线 (Roadmap)

*   **v0.1.0 (MVP 验证期 - 开源首发)：**
    *   实现核心底座：Agent 评审委员会（Prompt 链路拉通）。
    *   接入单一数据源：V2EX `qna` 和 `create` 节点。
    *   输出方式：终端直接打印 Markdown 格式立项书。
*   **v0.2.0 (生态扩展期)：**
    *   开放 Plugin 机制，允许社区贡献抓取工具（如：小红书监控插件、B站评论区分析插件）。
    *   增加本地 RAG 记忆库，防止 Agent 重复推送相同的商业点子。
*   **v1.0.0 (SaaS 化探索期)：**
    *   开发 Web UI，提供可视化的大盘看板。
    *   支持用户输入自己关心的行业关键词（如“牙医”、“二手车”），Agent 进行定向商业洞察。
